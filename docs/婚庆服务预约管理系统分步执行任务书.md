# 婚庆服务预约管理系统分步执行任务书

编制日期：2026年3月20日

关联文档：[婚庆服务预约管理系统项目规划书](./婚庆服务预约管理系统项目规划书.md)

## 一、文档说明

本任务书用于将《婚庆服务预约管理系统项目规划书》转化为可执行、可追踪、可验收的实施台账，供产品、研发、测试、运营在同一口径下协同推进。

执行约束如下：

- 本文档覆盖 `V1 / V1.5 / V2` 全路线图，但当前默认启动阶段为 `V1`。
- `V1` 聚焦主交易闭环，属于立即执行主线。
- `V1.5 / V2` 仅作为后续任务池，不应挤占 `V1` 核心资源。
- 任务状态、优先级、依赖、输出物、验收标准必须持续维护，不允许只更新口头进度。
- 阶段完成的判定以“模型、接口、页面、联调、验收闭环”为准，不以代码提交数量为准。
- 婚庆业务相关数据库改动统一通过 `server/sql/1.9.4.20260320/update.sql` 管理，不在 `server/public/install/db/like.sql` 中完成；后续所有涉及表结构、字段、索引、菜单初始化、配置初始化的任务默认都需同步到版本升级 SQL。

## 二、追踪规则

### 2.1 状态定义

| 状态 | 含义 |
| --- | --- |
| 未开始 | 尚未进入实际实施 |
| 进行中 | 已启动实施，有明确责任人与当前动作 |
| 已完成 | 已完成开发、联调与阶段验收 |
| 阻塞 | 因外部依赖或关键问题无法继续推进 |
| 已取消 | 经确认不再执行 |

### 2.2 优先级定义

| 优先级 | 含义 |
| --- | --- |
| P0 | `V1` 主链路关键任务，不完成则无法上线 |
| P1 | 重要支撑任务，不完成会影响效率、体验或可运营性 |
| P2 | 后续增强任务或优化项 |

### 2.3 责任端定义

| 责任端 | 说明 |
| --- | --- |
| `server` | ThinkPHP 后端模型、服务、接口、任务、事件 |
| `admin` | 管理后台页面、权限、运营配置、经营看板 |
| `uniapp` | 小程序/H5 用户端、服务人员中心、驾驶舱 |
| `跨端` | 同时涉及 `server + admin + uniapp` 的联动任务 |
| `运维配置` | 支付、订阅消息、企业微信、环境变量等部署侧工作 |
| `数据准备` | 行政区划、开放城市、模板内容、初始化配置等基础数据 |

### 2.4 字段说明

| 字段 | 说明 |
| --- | --- |
| 任务编号 | 采用 `Mx / Tx.y` 编码，便于引用与状态同步 |
| 所属阶段 | 任务所属里程碑阶段 |
| 前置依赖 | 启动前必须完成的任务或外部条件 |
| 输出物 | 该任务完成后可被验收的交付结果 |
| 验收标准 | 判断任务完成的客观标准 |
| 负责人 | 责任人占位，实施时补充到具体人员 |
| 预计工期 | 估算工期占位，实施时补充 |
| 实际完成时间 | 完成时补填 |

## 五、`V1` 阶段验收清单

### 5.1 主链路验收

- 用户只能选择已开放城市下的县区，并在选定档期后进入服务人员筛选。
- 服务人员列表页仅返回当前县区和日期下存在可售套餐的服务人员。
- 服务人员详情页仅展示当前县区可售套餐、匹配价格与价格命中层级。
- 用户可按后台配置的服务内容模板逐页填写内容并提交订单。
- 订单预览页可展示省市县地区、价格命中层级与最终价格。
- 订单提交后状态进入“待服务人员确认”。
- 服务人员接受后用户才进入支付或线下凭证流程。
- 在线支付成功或线下凭证审核通过后订单进入待履约。
- 履约完成后订单根据评价规则进入“待评价”“评价待审核”或“已完成”。

### 5.2 权限验收

- 普通用户仅可访问普通用户页面。
- 被绑定为服务人员的用户可进入服务人员中心。
- 被授权的用户可进入驾驶舱。
- 未授权用户不可直接访问驾驶舱与服务人员中心页面。
- 服务人员只能操作自己的资料、档期、订单、动态与评论审核数据。

### 5.3 地区、档期与订单一致性验收

- 同一服务人员同一自然日不允许产生冲突订单。
- 套餐价格按“县级 -> 市级 -> 省级”顺序正确匹配。
- 套餐无地区价格匹配时不展示，服务人员无任一可售套餐时不展示。
- 拒单、超时取消、支付超时、线下凭证驳回后档期可正确释放。
- 改期通过后旧档期释放、新档期占用。
- 后台调整地区价格后，历史订单快照中的地区与价格保持不变。

### 5.4 通知验收

- 主要业务节点均生成站内通知。
- 用户订阅消息在指定场景可正确触发。
- 服务人员企业微信消息在指定场景可正确触发。
- 推送失败不阻断主流程，且可在后台排查发送结果。

### 5.5 业务规则验收

- 开关关闭后，前端入口隐藏、按钮禁用、服务端写入拒绝同时生效。
- 仅开放的支付方式可在订单页展示与提交。
- 关闭评价后，履约完成订单按规则直接完成或进入自动完成队列。
- 关闭评论后，不允许新增评论，但历史已展示评论不被自动删除。
- 关闭动态发布后，不允许新增动态，但历史已通过内容保持展示。

### 5.6 审核权限验收

- `admin` 审核模式下，仅管理员可审核评价、动态、评论或资料变更。
- `provider` 审核模式下，服务人员仅可审核自己动态下的评论或自己的订单评价。
- `provider_then_admin` 模式下，初审与终审状态清晰可追踪。
- 服务人员资料、证书、作品、套餐、风格标签在审核通过前，线上保留旧版本。
- 未审核通过的评价、动态、评论不得公开展示，也不得进入经营统计。
