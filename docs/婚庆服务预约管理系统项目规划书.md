# 婚庆服务预约管理系统项目规划书

编制日期：2026年3月20日

## 一、项目背景

当前仓库基于通用的前后端分离管理框架搭建，现有结构包含：

- `admin`：Vue 3 + Vite 管理后台，已具备权限、菜单、素材、文章、消息通知、支付配置、退款记录、工作台等通用能力。
- `uniapp`：面向移动端的小程序/H5 应用，当前主要承载首页、资讯、个人中心、登录注册、支付结果等基础页面。
- `server`：ThinkPHP 6 后端，已具备用户体系、支付抽象、退款抽象、通知配置、配置中心、素材与文件上传等公共能力。
- `pc`：存量 Web 端壳层能力，本期不作为核心建设对象。

当前底座更偏通用内容/商城框架，尚不具备婚庆服务预约业务所需的服务人员、档期、服务订单、改期、评价、服务人员中心、经营驾驶舱等核心能力。

本项目目标是在现有框架基础上，二次开发为“婚庆服务预约管理系统”，重点建设服务人员单服务直购链路，并围绕预约场景补齐后台运营、评价、通知、服务人员中心、驾驶舱等能力。

## 二、项目目标

本期聚焦建立完整的婚庆服务预约主链路，主链路定义为：

`选档期与地区 -> 选服务人员 -> 多页面选择服务内容 -> 下单 -> 服务人员确认订单 -> 支付/线下凭证 -> 履约 -> 评价`

本期建设目标如下：

- 以“服务人员”为核心售卖单元，支持司仪、摄影、管家、督导等服务分类。
- 支持以地区、档期为前置条件筛选可服务人员，用户地区选择要求到县级，套餐价格按“县级优先、市级次之、省级兜底”匹配，降低无效咨询与无效下单。
- 支持管理员在后台维护平台可服务城市，精确到市级，并在套餐内配置服务地区与地区价格。
- 支持管理员在后台按服务分类配置“多页面服务内容选择模板”，用于用户下单前逐步采集服务需求。
- 支持用户在下单前维护婚礼基础档案，用于沉淀婚期、宴会城市、预算、风格偏好、联系人等通用信息，减少重复填写。
- 支持服务人员确认接单后再进入支付或线下凭证流程，贴合预约制服务业务。
- 支持在主交易链路稳定后补充咨询留资、收藏、候补提醒、开档提醒等转化能力，承接婚庆业务长决策周期场景。
- 支持分阶段支付能力，优先规划“定金 + 尾款”模式，用于适配婚庆高客单交易习惯。
- 支持服务人员通过同一小程序中的服务人员中心管理资料、作品、套餐、档期、订单与动态。
- 支持管理员通过 Web 后台进行统一运营配置、审核与经营管理，并通过小程序驾驶舱查看核心经营数据，经营口径需替换当前演示型数据。
- 对订单关键节点的相关人员发送站内通知；站外对用户发送小程序订阅消息，对服务人员推送企业微信消息。
- 支持为通知通道补齐发送日志、失败重试、人工补发等可靠性能力。
- 弱化现有通用装修系统，改为固定业务页 + 轻配置模式。
- 保留 `pc` 端并将其收敛为品牌获客层，重点承接城市页、案例页、服务人员详情页等内容型页面，不承担后台式复杂交易操作。

## 四、当前框架评估

### 4.1 可复用能力

#### 后台通用能力

- 后台账号、角色、菜单、权限控制已具备。
- 素材中心、图片上传、附件存储已具备。
- 字典、系统配置、支付配置、短信配置、小程序配置已具备。
- 消息通知设置页已具备基础管理入口。
- 工作台、财务、退款记录、日志、定时任务已具备基础模块。

#### 移动端基础能力

- 用户登录注册、个人中心、支付结果页已具备基础页面。
- 现有 `uniapp` 已形成统一请求、鉴权、支付封装，可复用。
- 现有小程序配置能力可为订阅消息与支付回调提供基础支撑。

#### 后端公共能力

- 已存在用户模型、支付逻辑、退款逻辑、通知记录模型、配置服务、文件服务。
- 已有支付方式配置、支付回调入口、退款日志与退款记录抽象。
- 已有通知设置表与站内通知记录表，可复用为婚庆业务通知底座。
- 已有事件与监听机制，可用于挂接业务通知发送。
- 已有定时任务能力，可用于处理确认超时、支付超时、提醒任务等。

### 4.2 现有不足

- 当前没有“服务人员”业务实体。
- 当前没有平台开放地区、套餐地区价格、档期、服务订单、订单快照、改期、评价、动态等业务模型。
- 当前没有婚礼基础档案、咨询留资、收藏候补、分阶段支付、订单凭证中心等预约业务增强模型。
- 当前通知设置虽支持 `system / sms / oa / mnp` 配置结构，但实际业务发送逻辑主要落在短信与站内记录上，企业微信消息尚未实现。
- 当前装修系统面向内容/商城首页，不适合作为预约下单的主业务承载。
- 当前工作台为演示型口径，不满足婚庆业务真实经营分析需求。
- 当前缺少适合婚庆行业的线索沉淀、城市运营和服务人员治理指标。

### 4.3 改造原则

- 复用公共能力，不重复造轮子。
- 交易与业务页面固定开发，不继续依赖通用装修页拼装。
- 新增业务配置采用独立业务模型，不混入通用装修系统。
- 婚庆业务开发期的数据库增量变更统一写入 `server/sql/1.9.4.20260320/update.sql`，包括表结构、字段、索引、初始化菜单、初始化配置等内容；本轮实施不在 `server/public/install/db/like.sql` 中完成相关改动。
- 先打通主交易链路，再补齐互动、运营与数据分析能力。
- 兼顾婚庆行业“高客单、长决策链路、强咨询前置”的业务特征，避免只建设可下单链路而忽略转化承接。

## 五、角色与端侧定位

### 5.1 普通用户

普通用户通过同一 `uniapp` 端完成以下操作：

- 维护婚礼基础档案
- 选择县级地区与档期
- 浏览并筛选服务人员
- 按多页面模板选择服务内容
- 咨询留资、收藏服务人员、提交候补提醒
- 提交订单
- 查看订单状态
- 在线支付或上传线下支付凭证
- 支付尾款
- 发起改期
- 完成后评价
- 接收站内通知与订阅消息

### 5.2 服务人员

服务人员不使用独立登录页，而是在后台创建服务人员资料时绑定一个已有用户账号。

判定规则：

- 后台新增服务人员时必须绑定 `user_id`。
- 登录用户若命中已绑定的服务人员资料，则在小程序“我的”页展示“服务人员中心”入口。
- 服务人员中心的数据权限只允许访问与自身绑定服务人员相关的数据。

服务人员中心支持：

- 资料维护
- 证书管理
- 作品管理
- 套餐管理
- 档期维护
- 订单确认与订单查看
- 动态发布
- 评论管理

### 5.3 驾驶舱查看用户

管理员小程序驾驶舱通过后台配置“可查看用户”实现，不做独立登录页。

判定规则：

- 后台配置可查看驾驶舱的用户。
- 登录用户命中后，在小程序展示“经营驾驶舱”入口。
- 驾驶舱默认只读，仅展示经营数据与待办摘要，不承担完整后台写操作。

### 5.4 平台管理员

平台管理员继续通过 `admin` 端完成：

- 服务资源管理
- 配置管理
- 业务规则配置
- 订单审核与异常处理
- 线下支付凭证审核
- 改期干预
- 评价管理
- 动态与评论审核
- 服务人员资料变更审核
- 通知模板与通道配置
- 经营分析与驾驶舱权限配置

## 六、整体业务方案

### 6.1 业务核心模式

本系统采用“服务人员预约制单服务直购”模式：

- 一个服务人员对应若干可售服务套餐。
- 平台后台统一配置可服务城市，精确到市级。
- 用户下单前先确定县级地区与档期。
- 套餐地区价格支持省、市、县三级配置，按“县级 -> 市级 -> 省级”顺序匹配。
- 用户完成服务内容选择后提交订单。
- 服务人员先确认接单，确认后用户再支付。
- 支付完成后进入履约阶段。
- 履约完成后进入评价阶段。

### 6.2 页面策略

本期不再以通用装修页面作为业务承载，统一改为固定业务页。

保留的轻配置能力包括：

- 首页轮播图
- 推荐位
- 服务分类排序
- 系统主题风格
- 小程序首页入口位

### 6.3 多页面服务内容配置策略

“多页面选择服务内容”采用独立业务配置方案，不使用现有装修系统。

配置粒度固定为“按服务分类配置”：

- 例如司仪一套模板，摄影一套模板，督导一套模板。
- 每个模板由多个页面组成。
- 每个页面由多个表单字段组成。
- 同分类下的服务人员默认使用同一套模板。
- `V1` 不做按服务人员单独覆盖。

字段类型建议包括：

- 单选
- 多选
- 文本输入
- 数字输入
- 长文本

每个模板字段需支持：

- 字段标题
- 字段说明
- 是否必填
- 选项列表
- 展示顺序
- 默认值

用户填写结果在订单提交时写入订单快照，后续流程不回读模板实时数据，避免模板变更影响历史订单。

### 6.4 婚礼档案与转化承接策略

婚庆业务具备“高客单、长决策周期、强咨询前置”的明显特征，因此系统不能只提供直接下单链路，还需要提供用户档案与线索承接能力。

策略如下：

- 用户在预约前可维护婚礼基础档案，沉淀婚期、预算、风格、场地、桌数、联系人等通用信息。
- 婚礼基础档案与订单服务内容模板分层管理：通用信息进入用户档案，服务专属信息进入订单模板与订单快照。
- 当城市未开放、服务人员无档期、用户暂不准备直接下单时，可转为咨询留资或候补提醒。
- `pc` 端优先承担品牌曝光与内容承接，不直接扩展为复杂交易主链路。

## 七、业务模块规划

### 7.1 服务资源域

#### 服务分类

后台维护婚庆服务分类，例如：

- 司仪
- 摄影
- 摄像
- 管家
- 督导

#### 风格标签

后台维护风格标签，例如：

- 中式
- 西式
- 轻奢
- 幽默
- 大气稳重

#### 服务人员档案

服务人员档案需要覆盖：

- 基础信息
- 头像与封面
- 个人介绍
- 所属服务分类
- 风格标签
- 从业年限
- 推荐位状态
- 上下架状态
- 绑定用户
- 企业微信接收标识

#### 证书与作品

- 证书支持上传图片、名称、说明、排序。
- 作品支持多图、封面、标题、说明、排序、显示状态。

#### 套餐

套餐用于定义可售卖单元，建议包含：

- 套餐名称
- 套餐简介
- 服务时长
- 适用服务分类
- 状态
- 地区价格表

套餐地区价格表支持：

- 省级价格
- 市级价格
- 县级价格
- 启用状态与排序
- 同一套餐下同一层级同一地区唯一配置

套餐可售规则：

- 用户前端必须先选择到县级地区。
- 管理员在后台先配置平台可服务城市，精确到市级。
- 套餐价格按“县级 -> 市级 -> 省级”顺序向上匹配。
- 未匹配到地区价格的套餐不展示。
- 服务人员在当前地区若无任一可售套餐，则不展示。

#### 平台开放地区

- 后台维护可服务城市，精确到市级。
- 前端地区选择器展示到县级，只允许选择已开放城市下的县区。
- 平台开放地区只决定是否可售，不直接决定套餐价格。

#### 推荐位与轮播图

- 首页轮播图由后台配置。
- 推荐服务人员、热门服务人员、精选案例等由后台配置推荐位。

### 7.2 地区与档期域

地区与档期能力包含：

- 平台开放城市管理
- 前端地区选择到县级
- 服务人员档期日历
- 后台统一查看档期
- 服务人员中心自维护档期
- 待确认锁档
- 已确认占档
- 冲突校验

地区规则包括：

- 用户必须选择县级地区后才可进入服务人员筛选。
- 未落在平台开放城市下的县区不可下单。
- 地区与档期共同作为服务人员与套餐的可售校验条件。

档期状态建议包含：

- 可预约
- 已锁定
- 已占用
- 不可服务

档期模型约束：

- 本期档期模型继续按“自然日”管理，不拆分为午宴、晚宴、全天、时段或缓冲场次。
- 档期冲突校验以“服务人员 + 服务日期”为最小冲突单元。
- 如后续业务增长需要引入按场次管理，应作为独立升级课题推进，不在当前版本主链路中实现。

档期占用规则：

- 用户提交订单后进入待确认锁档。
- 服务人员拒绝、订单超时取消、支付失败超时、线下凭证驳回时释放档期。
- 服务人员接单并支付成功后进入已占用。

### 7.3 用户档案与转化域

用户档案与转化能力包含：

- 婚礼基础档案
- 咨询留资
- 收藏服务人员
- 候补提醒
- 开档提醒
- 城市开放提醒

#### 婚礼基础档案

婚礼基础档案建议包含：

- 婚礼日期
- 宴会城市
- 宴会场地
- 桌数规模
- 预算区间
- 婚礼风格偏好
- 联系人
- 联系方式
- 备注信息

设计原则：

- 用户档案用于复用通用信息，不替代订单模板。
- 订单创建时可读取用户档案默认值，但订单快照仍以用户提交当时内容为准。
- 用户档案更新不回溯历史订单与历史咨询单。

#### 咨询留资与候补

- 当用户尚未准备直接下单时，可提交咨询留资，由平台或服务人员后续跟进。
- 当所选档期无可售服务人员时，可提交候补提醒。
- 当用户关注的城市暂未开放时，可提交城市开放提醒。
- 收藏、候补、提醒均需记录来源地区、服务分类、目标日期与用户联系方式，用于后续转化分析。

### 7.4 交易域

交易能力包含：

- 订单预览
- 创建订单
- 服务人员确认订单
- 用户取消订单
- 在线支付
- 线下支付凭证
- 支付结果处理
- 分阶段支付
- 退款
- 订单状态流转
- 订单快照

订单状态建议定义为：

- 待服务人员确认
- 服务人员已拒绝
- 待支付
- 待线下凭证审核
- 待履约
- 履约中
- 待评价
- 评价待审核
- 已完成
- 已取消
- 退款中
- 已退款

服务人员确认阶段权限固定为：

- 仅接受或拒绝
- 不允许改价
- 不允许改服务内容

#### 分阶段支付

为适配婚庆服务高客单和预约制场景，后续增强阶段建议补充分阶段支付能力，优先支持“定金 + 尾款”模式。

建议规则：

- `V1` 仍以单次支付或线下凭证支付为主。
- `V1.5` 开始规划分阶段支付，至少支持首款金额、尾款金额、尾款截止时间、尾款支付状态。
- 尾款提醒、尾款逾期、尾款补缴记录均需纳入订单状态追踪与通知体系。

### 7.5 订单变更域

本期支持改期，不支持复杂订单改价或内容重签。

改期流程包括：

- 用户发起改期申请
- 校验新档期是否可预约
- 服务人员确认通过或拒绝
- 平台后台可查看与干预

改期通过后：

- 释放旧档期
- 占用新档期
- 更新订单服务时间

内容互动包含：

- 服务人员动态发布
- 动态评论
- 订单评价

动态模块用于服务人员做内容运营，例如：

- 服务案例展示
- 工作花絮
- 履约现场分享

评价模块用于订单履约后沉淀口碑数据，支持：

- 评分
- 文字评价
- 图片评价

### 7.7 通知与经营域

通知包含：

- 站内通知
- 用户小程序订阅消息
- 服务人员企业微信消息

经营域包含：

- Web 后台经营看板
- 小程序驾驶舱核心数据
- 订单、退款、改期、评价等统计能力

经营指标建议补充：

- 预约漏斗数据，例如访问、筛选、咨询、下单、接单、支付、完成
- 服务人员经营数据，例如接单率、拒单率、退款率、评价得分、响应时长
- 城市运营数据，例如已开放城市数、城市供给覆盖率、可售套餐数、成交城市排行
- 履约与审核数据，例如线下凭证审核时效、评价审核时效、改期处理时效

通知与经营底座建议补充：

- 通知发送日志
- 通知失败重试
- 人工补发入口
- 业务操作日志
- 订单凭证中心

工作台与驾驶舱升级要求：

- 工作台不得继续使用演示型固定值或随机值。
- 所有核心经营指标必须基于真实业务数据生成。
- 驾驶舱保持只读，但需具备真实待办与趋势分析能力。

### 7.8 业务规则配置域

为避免审核、互动、支付、通知等规则散落在各业务模块中，平台后台需新增统一“业务规则配置中心”。

配置粒度约束为：

- 本期所有规则均为平台级全局配置。
- 不按服务分类、城市、服务人员做差异化配置。
- 核心交易约束仍保持固定，不允许被后台改写。

规则类型分为：

- 开关型：控制某项能力是否开启。
- 模式型：控制某项能力由谁审核、是否直发、是否需审批。
- 时效型：控制超时关闭、自动完成、提醒触发时间。

建议纳入的交易规则包括：

- `booking_enabled`：是否开放预约入口
- `lead_submit_enabled`：是否开放咨询留资入口
- `waitlist_enabled`：是否开放候补提醒入口
- `pay_online_enabled`：是否允许在线支付
- `pay_offline_voucher_enabled`：是否允许线下凭证支付
- `staged_payment_enabled`：是否开启分阶段支付
- `offline_voucher_audit_required`：线下凭证是否必须审核
- `user_cancel_order_enabled`：用户是否可主动取消订单
- `user_reschedule_enabled`：用户是否可发起改期
- `refund_apply_enabled`：用户是否可发起退款申请
- `auto_close_unconfirmed_enabled`：未接单订单是否自动关闭
- `auto_close_unconfirmed_minutes`：未接单订单自动关闭时长
- `auto_cancel_unpaid_enabled`：待支付订单是否自动取消
- `auto_cancel_unpaid_minutes`：待支付订单自动取消时长
- `auto_finish_after_service_enabled`：履约后未评价订单是否自动完成
- `auto_finish_after_service_days`：履约后自动完成天数

建议纳入的审核规则包括：

- `order_review_enabled`：是否允许订单评价
- `order_review_audit_mode`：订单评价审核模式，取值建议为 `none / admin / provider / provider_then_admin`
- `provider_post_enabled`：是否允许服务人员发布动态
- `provider_post_audit_mode`：动态发布审核模式，取值建议为 `none / admin`
- `post_comment_enabled`：动态是否允许评论
- `post_comment_audit_mode`：动态评论审核模式，取值建议为 `none / provider / admin`
- `provider_profile_edit_mode`：服务人员资料修改模式，取值建议为 `direct / admin_audit`
- `provider_certificate_edit_mode`：服务人员证书修改模式，取值建议为 `direct / admin_audit`
- `provider_work_publish_mode`：服务人员作品发布模式，取值建议为 `direct / admin_audit`
- `provider_package_edit_mode`：服务人员套餐修改模式，取值建议为 `direct / admin_audit`
- `provider_tag_edit_mode`：服务人员风格标签修改模式，取值建议为 `disabled / direct / admin_audit`

建议纳入的通知与展示规则包括：

- `system_notice_enabled`：是否开启站内通知
- `mnp_notice_enabled`：是否开启用户小程序订阅消息
- `wecom_notice_enabled`：是否开启服务人员企业微信消息
- `service_reminder_enabled`：是否开启履约前提醒
- `review_reminder_enabled`：是否开启待评价提醒
- `dashboard_enabled`：是否开放小程序驾驶舱入口

默认策略建议为：

- 预约开启、在线支付开启、线下凭证支付开启
- 线下凭证默认需要后台审核
- 改期、退款默认开启，但受订单状态约束
- 订单评价默认开启，且默认由管理员审核
- 服务人员动态默认开启，且默认由管理员审核后发布
- 动态评论默认开启，且默认由服务人员审核自己动态下的评论
- 服务人员修改风格标签默认需管理员审核
- 履约前提醒、待评价提醒、站内通知默认开启
- 用户订阅消息、企业微信消息是否开启取决于对应通道配置是否完成

明确不纳入开关的固定约束包括：

- 用户地区必须选择到县级
- 一个订单仅对应一个服务人员和一个服务套餐
- 服务人员确认接单后用户才进入支付
- 套餐价格按“县级 -> 市级 -> 省级”顺序匹配
- 订单快照生成后不回溯历史配置
- 驾驶舱保持只读，不承担后台完整写操作

## 八、核心流程设计

### 8.1 用户下单主流程

1. 用户进入服务预约入口，系统先校验 `booking_enabled` 是否开启。
2. 选择地区与档期，其中地区必须选择到县级。
3. 系统校验所选县区是否属于后台已开放城市，并按地区、档期、分类、标签筛选“当前地区存在可售套餐”的服务人员。
4. 用户进入服务人员详情页查看资料、作品、当前地区可售套餐与匹配价格。
5. 用户按后台配置的多页面模板逐步填写服务内容。
6. 系统展示订单预览，包含省、市、县地区、档期、服务人员、套餐、服务内容摘要、匹配价格层级与最终价格。
7. 用户提交订单，系统重新按“县级 -> 市级 -> 省级”计算套餐价格，生成订单与订单快照，固化地区与价格匹配结果，并锁定档期，订单状态为“待服务人员确认”。
8. 服务人员在服务人员中心确认接单或拒绝。
9. 若开启 `auto_close_unconfirmed_enabled`，订单在超时后自动关闭并释放档期。
10. 若确认接单，订单进入“待支付”。
11. 用户仅可选择后台已开启的支付方式：在线支付或线下支付凭证。
12. 若开启 `auto_cancel_unpaid_enabled`，待支付订单超时自动取消并释放档期。
13. 支付成功或线下审核通过后进入“待履约”。
14. 履约完成后，根据 `order_review_enabled` 与自动完成规则进入“待评价”或“已完成”。
15. 用户评价后，按评价审核规则进入“待审核”或直接进入“已完成”。

### 8.2 服务人员确认流程

1. 服务人员收到新订单通知。
2. 服务人员查看订单快照、时间、省市县地区、匹配价格层级、最终价格、服务内容摘要。
3. 服务人员选择“接受”或“拒绝”。
4. 接受后订单进入“待支付”。
5. 拒绝后订单进入“服务人员已拒绝”，系统释放档期并通知用户。
6. 若长时间未处理且开启自动关闭规则，系统自动关闭订单并通知双方。

### 8.3 线下支付凭证流程

1. 仅当 `pay_offline_voucher_enabled` 开启时，用户可选择线下支付方式。
2. 用户提交支付凭证图片与备注。
3. 若 `offline_voucher_audit_required=1`，订单进入“待线下凭证审核”。
4. 若 `offline_voucher_audit_required=0`，凭证提交后可直接进入“待履约”，后台仅保留补录与抽检能力。
5. 审核通过后进入“待履约”。
6. 审核驳回后退回“待支付”，并通知用户重新处理。

### 8.4 退款流程

退款能力复用现有退款底座，新增服务订单场景。

支持：

- 在线支付原路退款
- 线下退款记录登记

退款后要求：

- 写入退款记录
- 写入退款日志
- 更新订单退款状态
- 发送站内与站外通知

退款流程补充规则：

1. 仅当 `refund_apply_enabled` 开启时，用户端展示退款申请入口。
2. 平台后台仍保留手动发起退款与异常兜底处理能力。
3. 关闭退款申请入口只影响用户主动申请，不影响后台对异常订单执行退款。

### 8.5 改期流程

1. 仅当 `user_reschedule_enabled` 开启时，用户在可改期状态下提交新档期申请。
2. 系统校验新档期是否可用。
3. 生成改期记录。
4. 服务人员确认通过或拒绝。
5. 通过则替换档期并同步订单。
6. 拒绝则维持原订单不变。

### 8.6 评价流程

1. 仅当 `order_review_enabled` 开启时，履约完成后订单进入“待评价”。
2. 用户提交评分、文字、图片。
3. 系统根据 `order_review_audit_mode` 生成评价记录并判定状态：
4. `none`：评价直接生效并公开展示，订单进入“已完成”。
5. `admin`：评价进入后台待审核，订单进入“评价待审核”，由管理员终审。
6. `provider`：评价进入服务人员待审核，订单进入“评价待审核”，由服务人员审核自己的订单评价，管理员保留兜底处理权限。
7. `provider_then_admin`：评价先由服务人员初审，再由管理员终审，订单在终审完成前保持“评价待审核”。
8. 审核通过后订单进入“已完成”；审核驳回后评价退回修改或按平台规则处理。
9. 仅审核通过的评价参与服务人员详情页展示与经营统计。
10. 若关闭评价能力且开启自动完成规则，订单在指定时间后自动进入“已完成”。

### 8.7 动态与评论流程

1. 仅当 `provider_post_enabled` 开启时，服务人员可发布动态。
2. 动态发布后根据 `provider_post_audit_mode` 判定是否直接展示或进入后台待审核。
3. 仅当 `post_comment_enabled` 开启时，普通用户可对动态发表评论。
4. 评论提交后根据 `post_comment_audit_mode` 进入以下流程：
5. `none`：评论直接展示。
6. `provider`：评论进入待审核，由动态所属服务人员审核。
7. `admin`：评论进入后台待审核，由管理员审核。
8. 未审核通过的动态与评论不对外展示，不进入统计口径。

### 8.8 服务人员资料变更流程

1. 服务人员在中心修改资料、证书、作品、套餐、风格标签时，系统读取对应修改模式。
2. `direct`：修改后直接生效。
3. `admin_audit`：生成变更申请，当前线上生效数据保持不变，待管理员审核通过后再切换。
4. `disabled`：前端不展示编辑入口，服务端拒绝写入。
5. 管理员可在后台查看变更前后差异、审核记录、驳回原因与生效时间。

### 8.9 婚礼档案与咨询转化流程

1. 用户进入预约入口前，可先维护婚礼基础档案。
2. 用户在档案中填写婚期、城市、预算、风格、场地、桌数与联系人信息。
3. 当用户满足下单条件时，系统将婚礼基础档案默认带入咨询单或订单填写页。
4. 当用户尚未准备直接下单时，可提交咨询留资，由平台或服务人员后续跟进。
5. 当用户目标档期无可售服务人员时，可提交候补提醒。
6. 当用户目标城市暂未开放时，可提交城市开放提醒。
7. 后台需记录来源城市、服务分类、目标日期、用户状态与跟进结果，用于后续转化统计。

### 8.10 分阶段支付流程

1. 当 `staged_payment_enabled` 开启且订单命中分阶段支付规则时，订单在接单后先进入“待支付定金”。
2. 用户支付定金后，订单进入“待履约”或“待尾款”，具体以平台规则为准。
3. 履约前或指定时间点触发尾款提醒。
4. 用户支付尾款后，订单进入尾款已结清状态。
5. 尾款逾期需记录状态、催付记录与人工干预结果。
6. 分阶段支付场景下退款需区分定金、尾款与整单退款口径。

## 十、接口与能力边界

### 10.1 用户端接口

## 十四、验收标准

### 14.1 主链路验收

- 用户只能选择已开放城市下的县区，并可先选地区与档期，再筛选出可服务人员。
- 服务人员详情页仅展示当前县区可售套餐与匹配价格。
- 用户可按后台模板逐页填写服务内容并提交订单。
- 订单预览可展示省市县地区、价格命中层级与最终价格。
- 订单提交后状态进入“待服务人员确认”。
- 服务人员接受后用户才进入支付流程。
- 在线支付成功或线下凭证审核通过后订单进入待履约。
- 履约完成后订单根据评价规则进入“待评价”“评价待审核”或“已完成”。

### 14.2 权限验收

- 普通用户仅可访问普通用户页面。
- 被绑定为服务人员的用户可进入服务人员中心。
- 被授权的用户可进入驾驶舱。
- 未授权用户不可直接访问驾驶舱与服务人员中心相关页面。
- 服务人员只能操作自己的资料、档期、订单与动态。

### 14.3 档期、地区与订单一致性验收

- 同一服务人员同一档期不允许产生冲突订单。
- 套餐价格按“县级 -> 市级 -> 省级”顺序正确匹配。
- 套餐无地区价格匹配时不展示，服务人员无任一可售套餐时不展示。
- 拒单、超时取消、支付超时、线下审核驳回后档期可正确释放。
- 改期通过后旧档期释放、新档期占用。
- 下单后即使后台调整地区价格，历史订单快照中的地区与价格不回溯变更。

### 14.4 通知验收

- 所有主要节点生成站内通知。
- 用户订阅消息在指定场景触发。
- 服务人员企业微信消息在指定场景触发。
- 推送失败有日志且不阻断主流程。

### 14.5 转化与分阶段支付验收

- 用户可维护婚礼基础档案，后续咨询或下单时可自动带出默认值。
- 当用户暂不下单时，可提交咨询留资并被后台检索与跟进。
- 当目标档期无可售服务人员时，可提交候补提醒。
- 当目标城市未开放时，可提交城市开放提醒。
- 分阶段支付开启后，订单可正确流转定金、尾款与结清状态。
- 尾款提醒与尾款逾期状态可在后台与通知体系中追踪。

### 14.6 业务规则验收

- 开关关闭后，前端入口隐藏、按钮禁用、服务端拒绝写入三层同时生效。
- 仅开放的支付方式可在订单页展示并提交。
- 关闭评价后，履约完成订单按规则直接完成或进入自动完成队列。
- 关闭评论后，不允许新增评论，但历史已展示评论不被自动删除。
- 关闭动态发布后，服务人员端不再允许新增动态，但历史已通过内容保持展示。
- 未审核通过的评价、动态、评论不得公开展示，也不得进入经营统计。

### 14.7 审核权限验收

- `admin` 审核模式下，仅管理员可审核评价、动态、评论或资料变更。
- `provider` 审核模式下，服务人员仅可审核自己动态下的评论或自己的订单评价。
- `provider_then_admin` 模式下，初审与终审状态清晰可追踪。
- 服务人员资料、证书、作品、套餐、风格标签在审核通过前，线上展示内容保持为旧版本。
- 管理员始终保留兜底复核与强制处理权限。

## 十五、风险与依赖

### 15.1 主要风险

- 档期锁定与释放逻辑若设计不严谨，易出现并发冲突。
- 当前档期模型按天管理，若业务后续出现同日多场履约诉求，需提前评估升级成本。
- 模板配置与订单快照边界若不清晰，易导致历史订单展示不一致。
- 行政区划、开放城市、套餐地区价格三者关联若不清晰，易导致可售判断和价格命中错误。
- 套餐地区价格支持省、市、县三级配置，后台若缺少唯一性校验与提示，易出现运营误配置。
- 企业微信消息通道为新增能力，需补齐配置、日志、失败处理。
- 用户订阅消息依赖用户授权，实际触达率存在波动。
- 服务人员中心与驾驶舱共用同一小程序入口，需要做好权限与路由隔离。
- 咨询留资、候补提醒、分阶段支付引入后，后台跟进与财务口径需要同步规范。

### 15.2 关键依赖

- 小程序支付与订阅消息配置完整
- 企业微信应用配置完整
- 服务人员账号绑定规则明确
- 行政区划数据准备完整，至少覆盖到县级
- 平台开放城市配置规则明确
- 套餐地区价格配置规则明确
- 运营端梳理好各服务分类的多页面模板内容

## 十六、默认约束与实施假设

- 一个订单仅对应一个服务人员和一个服务套餐。
- 服务地区不在服务人员档案中维护，由“平台开放城市 + 套餐地区价格”共同决定。
- 用户地区选择必须到县级，前端只展示已开放城市下的县区。
- 档期模型当前按“自然日”管理，不拆分午宴、晚宴、全天或时段。
- 套餐价格按“县级 -> 市级 -> 省级”顺序匹配，若均未命中则不可售。
- 省级价格仅对该省内已开放城市下的县区生效。
- 订单快照固化下单时的省市县地区、价格命中层级与最终价格，后续配置变更不回溯历史订单。
- 婚礼基础档案用于复用用户通用信息，不替代订单模板与订单快照。
- 多页面服务内容模板按服务分类配置，不做服务人员级定制。
- 服务人员确认阶段仅允许接受或拒绝，不允许改价、改内容。
- 驾驶舱只读，不承担后台完整管理能力。
- 业务规则配置统一按平台级全局生效，不按服务分类、城市、服务人员做差异化配置。
- 订单评价默认开启，默认审核模式为管理员审核。
- 服务人员动态默认开启，默认审核模式为后台管理员审核后发布。
- 动态评论默认开启，默认审核模式为服务人员审核自己动态下的评论。
- 服务人员资料、证书、作品、套餐、风格标签修改默认走管理员审核。
- 线下支付凭证默认需要后台审核。
- 分阶段支付优先支持“定金 + 尾款”，不在当前版本扩展更复杂的多段收款方案。
- 关闭评价、评论、动态等能力时，仅禁止新增，不回收历史已生效内容。
- 核心交易约束不做成后台开关，包括县级地区必选、单订单单服务人员、价格三级匹配、订单快照不回溯。
- `pc` 端继续保留，但定位为品牌获客层，不重构为核心交易主链路。
- 本期不再继续扩展通用装修系统承载业务交易流程。

## 十七、结论

本项目建议以现有通用框架为底座，采用“固定业务页 + 可配置服务内容模板 + 统一业务规则配置中心 + 统一通知中台 + 多角色同端入口”的方式推进建设，并以“平台开放城市 + 套餐地区价格”作为地区能力设计基础。首要目标仍是打通“县级地区/按天档期筛选、服务人员选择、服务内容采集、订单确认、支付、履约、评价”主链路；在此基础上，再逐步补齐婚礼基础档案、咨询留资、候补提醒、分阶段支付、通知可靠性、真实经营分析以及 `pc` 品牌获客层能力，以最小成本完成从通用框架向婚庆预约业务系统的转型。
